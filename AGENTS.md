# Staymod â€” agent guide

Follow these conventions so the codebase stays consistent for humans and AI assistants.

## Project layout

| Area | Location | Purpose |
|------|----------|---------|
| Routes & layouts | `app/` | Next.js App Router pages, layouts, route handlers (`app/api/**/route.ts`). |
| Global UI | `components/global/` | Shell pieces (header, providers) imported from layouts and pages. |
| Design system | `components/ui/` | shadcn/ui primitives (generated by the shadcn CLI). |
| Hooks | `hooks/` | Client hooks; TanStack Query wrappers (`useApiQuery`, `useApiMutation`). |
| API surface (typed clients) | `api-clients/` | Functions that call backend routes or external HTTP APIs via `apiFetch` â€” **not** raw `fetch` scattered in components. |
| Utilities | `utils/` | Shared helpers: `api-fetch.ts`, `mongodb.ts`, server-only auth helpers. |
| Middleware helpers | `middlewares/` | Logic used with Clerk / Next middleware (the **default export** for the edge entry must stay in root `proxy.ts` on Next.js 16). |
| Types | `types/` | Shared TypeScript types (e.g. organization scoping). |
| shadcn `cn()` helper | `lib/utils.ts` | Required by shadcn components (`cn` from `class-variance-authority` + `tailwind-merge`). |

Add new shadcn components with `npx shadcn@latest add <name>` so `components.json` stays authoritative.

## MongoDB

- **Single connection**: use `getDb()` / `getMongoClient()` from `@/utils/mongodb` everywhere (Route Handlers, Server Actions, server-only modules). Do not create ad hoc `MongoClient` instances in feature code.
- **Environment**: `MONGODB_URI` is required when the database is used. Optional `MONGODB_DB_NAME` defaults to `staymod`.
- **Scope by organization**: when the active Clerk organization matters, resolve `orgId` with `getActiveOrganizationId()` from `@/utils/auth-org` and include it in queries (or map Clerk org id â†’ your own collections).

## HTTP and TanStack Query

- **All HTTP** goes through `apiFetch` in `@/utils/api-fetch` (JSON handling, errors as `ApiError`).
- **Typed route clients** live in `@/api-clients/*` (e.g. `fetchHealth()`), calling `apiFetch`.
- **Client components** use TanStack Query: prefer `useApiQuery` / `useApiMutation` from `@/hooks` so keys and `queryFn` stay consistent. You may use `useQuery` / `useMutation` directly when the `queryFn` is an `api-clients/` function â€” still must use `apiFetch` under the hood.

Do not call bare `fetch("/api/...")` in UI unless you extend `apiFetch` first for that case.

## Authentication and organizations (Clerk)

- **Provider**: `ClerkProvider` wraps the app in `app/layout.tsx`.
- **Middleware**: root `middleware.ts` exports `clerkMiddleware()` and an **inline** `config.matcher` array. Next.js 16 requires this matcher to be statically analyzable â€” do not move it to a shared imported constant.
- **Organizations**: enable Organizations in the Clerk dashboard. Use `OrganizationSwitcher` (see `components/global/app-header.tsx`) and `orgId` from `auth()` / `getActiveOrganizationId()` on the server. Treat Clerk as the source of truth for membership and roles.

## UI

- Use **shadcn/ui** components from `@/components/ui/*` and compose in `components/global` or feature folders.
- Clerk UI primitives (`UserButton`, `SignInButton`, etc.) come from `@clerk/nextjs`. Clerk v7 may not export older control components; use `useAuth()` for signed-in vs signed-out UI when needed.

## Environment

Copy `.env.example` to `.env.local` and fill in Clerk and MongoDB variables before running auth- or DB-backed features.
